# 数据库设计说明

## 1. 表关系

- `category` 与 `resource` 是一对多关系，一个分类下可以有多个资源。
- `resource` 与 `resource_preview` 是一对多关系，一个资源可配置多张预览图。
- `resource` 与 `tag` 是多对多关系，通过 `resource_tag` 关联。
- `resource` 与 `resource` 的相关推荐关系通过 `resource_recommend` 维护。
- `faq` 作为全站 FAQ 数据源，首页和详情页按配置或业务规则取用。
- `lead` 独立记录咨询线索，不依赖前台用户注册体系。
- `site_config` 用键值对方式承载站点配置与首页配置。
- `page_content` 用于承载关于我们、联系我们、免责声明、退款说明等固定页面内容。

## 2. 设计理由

- 这是一个导流型 MVP，核心目标是“快速上线、可管理、可继续扩展”，因此优先采用单体数据库设计。
- `site_config` 采用键值对而不是拆出多张配置表，可以减少表数量，便于后台统一保存首页配置和站点配置。
- `content_items_json` 采用 JSON 数组字符串存储，适合“资源包含内容清单”这种结构固定、读取频繁、无需复杂联表查询的数据。
- `resource_preview` 单独拆表，便于按排序维护多张预览图，也方便未来扩展视频预览。
- `resource_recommend` 单独拆表，便于运营在后台灵活配置“相关资源推荐”。
- 本版数据库不强制添加外键约束，关系由应用层维护，原因是项目采用逻辑删除，后台运营修改和数据初始化会更灵活。

## 3. `content_items_json` 示例

```json
["课件PPT","配套教案","逐字稿","课堂互动页","家校通知模板"]
```
