# Docker Compose 封装设计

## 目标

为“小学课件教案资源导流站”补齐一套可直接启动的 Docker Compose 环境，使项目在本地无需分别安装和启动 MySQL、后端服务、前端服务即可运行，并支持启动后的真实联调验证。

## 方案选择

### 方案 A：MySQL + Spring Boot Jar + Nginx 静态前端

这是本次采用的方案。

- 优点：最接近正式部署形态，容器职责清晰，启动后验证结果更可信。
- 优点：前端通过 Nginx 统一托管静态文件并反向代理 `/api`，浏览器侧无需额外处理跨域。
- 优点：后端保持标准 Spring Boot Jar 启动方式，后续迁移到服务器或云主机也更顺滑。

### 方案 B：MySQL + Spring Boot + Vite 开发服务器

- 优点：前端热更新更方便。
- 缺点：更偏开发态，不适合作为“正式交付环境”。

### 方案 C：仅数据库容器化

- 优点：最轻。
- 缺点：不符合“把整个环境封装成 docker compose”的目标。

## 容器组成

- `mysql`
  - 使用 `mysql:8.4`
  - 通过挂载 `sql/` 目录自动完成建表、索引和种子数据初始化
  - 容器内部端口 `3306`，宿主机映射 `3307`

- `backend`
  - 使用多阶段构建，基于 `maven:3.9.9-eclipse-temurin-21` 打包
  - 运行阶段使用 `eclipse-temurin:21-jre`
  - 通过环境变量覆盖数据库连接与 JWT 配置
  - 暴露宿主机端口 `8080`

- `frontend`
  - 使用 `node:20-alpine` 构建 React 静态资源
  - 使用 `nginx:1.27-alpine` 提供静态站点
  - `Nginx` 将 `/api` 反向代理到 `backend:8080`
  - 宿主机映射端口 `5173`

## 关键取舍

- 不额外引入网关容器，避免 MVP 阶段容器职责过多。
- MySQL 宿主机端口使用 `3307`，减少与开发机已有 MySQL `3306` 冲突的概率。
- 前端容器不运行 Vite dev server，而是运行 Nginx 静态站点，保证 Docker 环境更稳定。
- 后端 `application.yml` 改为支持环境变量覆盖，保持本地直跑与容器运行兼容。

## 验证方式

启动后进行三层验证：

1. 容器状态验证：确认三个服务都已启动。
2. 接口验证：请求首页接口、资源列表接口、页面内容接口、后台登录接口。
3. 页面验证：请求前端首页 HTML，并确认前端可通过 Nginx 正常访问。
